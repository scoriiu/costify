Fluxul contabil în România — pentru programatori

De la documentul justificativ la situațiile financiare, cu detalii tehnice de implementare, structuri de date, algoritmi de parsare și edge cases.

01 Imaginea de ansamblu

Gândește-te la contabilitate ca la un ETL pipeline (Extract-Transform-Load). Datele brute (documentele) intră în sistem, trec prin transformări succesive, și la final se produc rapoarte standardizate.

Input

Document justificativ

Factură, chitanță, extras de cont — source_id al fiecărei tranzacții.

Transform 1

Notă contabilă (articol contabil)

Traducerea documentului în limbaj contabil: conturi + sume. Partidă dublă: debit == credit.

Storage — Single Source of Truth

Registrul Jurnal

Toate notele contabile, în ordine cronologică. Baza de date brută — din el se pot reconstitui toate celelalte.

Transform 2 — GROUP BY cont

Cartea Mare / Fișe de cont

Același dataset, altă perspectivă: mișcările grupate pe fiecare cont.

Transform 3 — SUM() + GROUP BY

Balanța de verificare

Sinteza tuturor conturilor: solduri inițiale, rulaje, solduri finale. „Tabloul de bord" al contabilității.

Output

Situații financiare

Bilanț

Active = Datorii + Capital

CPP

Venituri − Cheltuieli = Rezultat

Alte rapoarte

Cash-flow, KPI, note

Regula fundamentală
Registrul Jurnal este cel mai granular. Dacă ai acces la el, poți reconstitui totul: balanță, bilanț, CPP, KPI.

02 Documentul justificativ

Orice document (hârtie sau electronic) care dovedește că s-a întâmplat ceva din punct de vedere economic.

Exemple: Factura, chitanța, extrasul de cont bancar, bonul fiscal, statul de plată, procesul-verbal de recepție.

Regulă de aur
Nicio înregistrare contabilă nu se face fără un document justificativ. Gândește-te la el ca la un source_id — fiecare rând din jurnal trebuie să aibă o referință la documentul care l-a generat.

03 Nota contabilă

„Traducerea" documentului în limbajul contabilității. Principiul partidei duble: fiecare operațiune afectează minimum două conturi — unul se debitează, altul se creditează. Suma debitului = suma creditului, întotdeauna.

Gândește-te la partidă dublă ca la o tranzacție bancară: când transferi bani, contul tău scade (credit) și contul destinatarului crește (debit). Suma totală din sistem rămâne aceeași.

Exemplu: Achiziție laptop 5.000 lei + TVA 19%

Cont debitatCont creditatSumă
214 Mobilier, aparatură404 Furnizori imobilizări5.000 lei
4426 TVA deductibilă404 Furnizori imobilizări950 lei
 
// Traducere în cod
debit(214, 5000)   // crește valoarea echipamentelor
credit(404, 5000)  // crește datoria către furnizor

debit(4426, 950)   // crește TVA de recuperat
credit(404, 950)   // crește datoria către furnizor
Un document → mai multe note contabile
Laptopul va genera lunar o notă de amortizare: 6811 = 2814 cu ~139 lei/lună, timp de 36 de luni. Deci 1 factură = 1 notă achiziție + 36 note amortizare.

04 Registrul Jurnal — structură și procesare

„Baza de date" a contabilității. Fiecare rând = o mișcare contabilă atomică: un cont debitat, un cont creditat, o sumă.

4.1 Structura unui rând (JournalEntry)

interface JournalEntry {
  // Identificare temporală
  data: Date;              // Data operațiunii
  year: number;            // Anul (extras din data)
  month: number;           // Luna (extrasă din data)

  // Referință document
  ndp: string;             // Nr. document primar (ex: "FV-1234")

  // Conturile afectate
  contD: string;           // Cont DEBIT complet (ex: "401.00023")
  contDBase: string;       // Cont DEBIT bază (ex: "401")
  contC: string;           // Cont CREDIT complet (ex: "5121.00001")
  contCBase: string;       // Cont CREDIT bază (ex: "5121")

  // Valoarea
  suma: number;            // Sumă în RON (întotdeauna pozitivă)
  explicatie: string;      // Descrierea operațiunii

  // Opționale
  categorie: string | null;
  cod: string | null;        // Cod partener (CUI, cod intern)
  tip: string | null;        // "S" = storno/extrabilan
}

4.2 Cont complet vs. cont bază

Distincția critică pentru procesare:

contD = "401.00023"    →  contDBase = "401"
contC = "5121.00001"   →  contCBase = "5121"
contD = "6811"         →  contDBase = "6811"     // fără punct = e deja bază
contD = "4111.ROCHE"   →  contDBase = "4111"   // poate fi și text după punct

Regula: Tot ce e ÎNAINTE de punct = contul sintetic (bază). Tot ce e DUPĂ punct = contul analitic (detaliu). Baza determină tipul contului; analiticul determină partenerul sau detaliul.

06 Nota contabilă compusă (compound entries)

Una din cele mai complicate părți ale parsării. Un cont e FIX (același pe mai multe rânduri), celălalt variază. Se marchează cu % pe rândul-antet.

Exemplu — factură cu 3 poziții

// În jurnal — notă compusă cu credit fix:

Rând 2 (DETALIU):  contD = "626"   contC = ""            suma = 200.00
Rând 3 (DETALIU):  contD = "626.1" contC = ""            suma = 150.00
Rând 4 (DETALIU):  contD = "4426"  contC = ""            suma = 66.50

07 Închiderea de lună și detecția post-closing

La sfârșitul lunii, contabilul face o „închidere". În jurnal apare ca o înregistrare cu explicația "Închidere luna". Problema: după închidere pot apărea intrări cu dată din aceeași lună, care sunt de fapt ale lunii următoare.

Rând 100: 31.03.2024, "Închidere luna martie"

function getAccountType(contBase: string): "A" | "P" | "B" {
  if (contBase.startsWith("1")) {
    if (contBase === "129") return "A";  // excepție
    return "P";  // Clasa 1 — Capitaluri → PASIV
  }
  if (contBase.startsWith("2")) {
    if (contBase.startsWith("28") || contBase.startsWith("29")) return "P";
    return "A";  // Clasa 2 — Imobilizări → ACTIV
  }
  if (contBase.startsWith("3")) {
    if (contBase.startsWith("39")) return "P";
    return "A";  // Clasa 3 — Stocuri → ACTIV
  }
  if (contBase.startsWith("4")) return "B";  // Clasa 4 — Terți → BIFUNCȚIONAL
  if (contBase.startsWith("5")) {
    if (contBase === "519") return "P";
    return "A";  // Clasa 5 — Trezorerie → ACTIV
  }
  if (contBase.startsWith("6")) return "A";  // Cheltuieli → cresc pe debit
  if (contBase.startsWith("7")) return "P";  // Venituri → cresc pe credit
  return "A";
}

10 Calculul balanței de verificare din jurnal

Operația centrală: transformarea jurnalului (date brute) în balanță (date agregate).

function computeBalance(entries, year, month?) {
  // PASUL 1: Solduri inițiale (tot ce s-a întâmplat ÎNAINTE de anul selectat)
  opening = {}
  for entry in entries where entry.year < year:
    opening[entry.contDBase].debit += entry.suma
    opening[entry.contCBase].credit += entry.suma

  // IMPORTANT: La fiecare tranziție de an anterior,
  // conturile cls 6+7 se ÎNCHID — se transferă în 121

  // PASUL 2: Rulaje pe anul/luna selectat(ă)
  turnovers = {}
  for entry in entries where entry.year == year
                        AND (month == null OR entry.month == month):
    turnovers[entry.contDBase].debit += entry.suma
    turnovers[entry.contCBase].credit += entry.suma

  // PASUL 3: Generare rânduri de balanță
  for each cont:
    sold_initial = opening[cont]
    sold_final = sold_initial + rulaj_debit - rulaj_credit  // pt tip A
    emit { cont, sold_initial, rulaj_debit, rulaj_credit, sold_final }
}

11 Închiderea de an — conturile de rezultat

La 31 decembrie, conturile cls 6 (cheltuieli) și cls 7 (venituri) se golesc — soldurile se transferă în contul 121 "Profit și pierdere". Conturile de rezultat măsoară performanța pe un singur an — la început de an nou pornesc de la zero.

function closeYearEnd(balances) {
  let profit121_debit = 0, profit121_credit = 0;

  for (const [cont, bal] of balances) {
    if (cont.startsWith("6")) {         // Cheltuieli — sold debitor
      const net = bal.debit - bal.credit;
      if (net > 0) profit121_debit += net;
      bal.debit = 0; bal.credit = 0;    // resetare
    }
    if (cont.startsWith("7")) {         // Venituri — sold creditor
      const net = bal.credit - bal.debit;
      if (net > 0) profit121_credit += net;
      bal.debit = 0; bal.credit = 0;    // resetare
    }
  }
  // Transfer în 121
  balances.get("121").debit += profit121_debit;
  balances.get("121").credit += profit121_credit;
  // Rezultat: credit > debit → PROFIT; debit > credit → PIERDERE
}

23 Edge cases și cazuri speciale

  • Intrări extrabilanțiere (storno): tip = "S" — pot fi filtrate din calcule
  • Contul 121 în închideri: intrări cu 121 se exclud din CPP (dubla numărare)
  • Conturi cu prefix comun: 401 și 4011 sunt diferite — se folosește contBase
  • Luni fără activitate: rulaje = 0, sold final = sold inițial — nu e eroare
  • Ani cu date parțiale: primul an poate să nu înceapă din ianuarie
  • Analitice cu cod text: 401.ORANGE — parserul tratează tot ce e după punct ca analitic

24 Glosar rapid

DebitPartea stângă. Pentru conturi A/cheltuieli = crește. Pentru P/venituri = scade.
CreditPartea dreaptă. Invers față de debit.
RulajSUM() pe o perioadă — cât s-a mișcat (ca un delta).
SoldDiferența netă debit − credit. Ca un balance după toate tranzacțiile.
Partidă dublăFiecare tranzacție → min. 2 conturi; total debit = total credit. Tranzacție atomică în DB.
Cont sinteticContul standardizat (3-4 cifre). Ca un enum definit de stat.
Cont analiticSub-cont creat de firmă (sintetic + punct + cod). Ca un foreign key la detaliu.
BifuncționalCont care poate fi A sau P. Clasificarea se face la runtime, pe baza semnului soldului.
Închidere anGolirea conturilor cls 6+7 și transfer în 121. Ca un TRUNCATE + INSERT INTO 121.
Notă compusăO notă cu un cont fix și mai multe detalii. Se „explodează" în note simple la parsare.
